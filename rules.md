# Soul Symphony Project Rules

## Hard Rules

1. **No simplifications, demonstrations, or partial implementations.** Soul Symphony is a professional game. Every feature must be production-grade and complete. No stubs, no TODOs left behind, no "simplified for brevity."

2. **No unnecessary or AI-looking comments.** Comments and documentation must read like a human wrote them: concise, purposeful, not decorative. No AI-style narration, no corny remarks, no stating the obvious, no corporate jargon, and absolutely no decorative separators or dividers like `------------`.

3. **No AI attribution.** All credit goes to the author: **ecnivs**. No "generated by" or "assisted by" footprints anywhere in code, docs, or commits.

4. **New features must be heavily optimized.** Performance is not an afterthought. Every addition should be benchmarked, profiled, and tuned before being considered done.

5. **Clean up after yourself.** If development introduces dead code, redundant paths, or orphaned logic, remove it immediately. No cruft accumulates.

6. **Benchmarking uses hyperfine.** All performance benchmarks must be run through `hyperfine` for consistent, reproducible results.

7. **Brutal honesty over politeness.** No sugarcoating, no yes-man behavior. If a decision is wrong for the future of the game, say so directly and with strong opinions. The game's long-term health comes first.

8. **Vector-first asset pipeline.** All game assets — textures, geometry, UI elements, effects — must be stored and processed in vector formats (SVG, SVG paths, vector math descriptions). No raster assets unless procedurally generated from vectors at runtime. No baked raster textures checked into the repo.

9. **Procedural generation preferred.** Wherever possible, generate content procedurally rather than storing it. Environments, patterns, effects, variations, and repetitive elements should come from code. Authoring vectors is for the unique, authored core; everything else is algorithmic.

10. **Color cycling is mandatory for interactive objects.** Every interactive object in the game — buttons, collectibles, enemies, NPCs, UI highlights, interactive environment elements — must rapidly cycle colors through a defined palette. The palette is global and consistent. Non-interactive elements (static backgrounds, ambient decoration) do not cycle. Cycling speed and phase may vary per object but must stay within the palette.

11. **Multi-lingual from day one.** All user-facing text — UI elements, menus, dialog, tooltips, notifications, error messages — must live in dictionary files. Format: `en.dict`, `th.dict`, `ko.dict`, `ja.dict`, `zh.dict`. No hardcoded strings. No default fallback to English without explicit handling. Every language is first-class.

12. **Build for 2030.** Target hardware capabilities expected in 2030: high-core CPUs, ray tracing standard, NVMe as baseline, 16GB+ memory common, VR/AR convergence, latency-sensitive rendering. Optimize for future hardware without breaking on current mid-range. No legacy concessions for 2020-era constraints unless unavoidable.

13. **ling-lang where possible.** Write game logic, systems, tools, and build scripts in ling-lang unless there's a hard technical reason not to (e.g., engine bindings, platform SDKs, performance-critical inner loops proven to need lower-level code). The game should eventually be self-hosting in ling-lang.

14. **No oversized files.** Keep source files within a reasonable count of lines (ideally under 500 lines, and strictly not exceeding 1000 lines). Keep files focused, concise, and modular; do not let them become bloated. Split them into clean, separate files/modules when they exceed these limits or take on unrelated responsibilities.

15. **Zero warnings or errors.** The code must compile and lint perfectly. No warnings, errors, or lints are allowed to remain in active code.

16. **Tests and benchmarks must pass cleanly.** All unit tests, integration tests, and performance benchmarks must pass completely without any warnings, deprecations, or errors.

17. **No commented-out code or obsolete comments.** Never leave commented-out blocks of code, temporary debugging comments, or stale explanations in the codebase. Actively clean up unnecessary or redundant comments.

18. **Clean up temporary and one-use files.** Any scratchpads, mock files, one-use scripts, or experimental test files created during development must be deleted once they are no longer needed. Never commit temporary development artifacts.

19. **Write highly scalable and clean code.** Code quality and scalability are non-negotiable. Avoid short-sighted hacks, hardcoded limits, or monolithic designs. Design components to be modular, extensible, and clean from day one.

20. **Do not commit code unless explicitly instructed.** Never perform Git commits or run git commit commands on behalf of the user unless they have explicitly instructed you to do so.

21. **No cheating on optimizations.** Performance optimizations must be genuine, general-purpose improvements. Never optimize specifically for a target benchmark's inputs or parameters at the expense of other code paths, overall performance, or correctness.

22. **Cross-platform compatibility is mandatory.** The game must run on Windows, macOS, Linux, and Web (WASM/JS) from the same codebase. Never hardcode platform-specific assumptions, paths, or dependencies; always use dynamic detection or unified abstractions.

23. **Dictionary files are the master record for text.** Every string used in the game must have an entry in all dictionary files. Missing entries are a build error. Dictionary files must be complete, precise, and kept in sync. Key naming convention: `ui.menu.start`, `dialog.npc.greeting.001`, `error.invalid_input`, etc.

24. **Palette is globally defined.** The color cycling palette is defined once in a single source file. All interactive objects reference this palette. No hardcoded hex values scattered across the codebase. Palette changes propagate universally.

25. **Procedural generation must be deterministic.** Same seed produces same output across all platforms. No platform-specific floating-point drift. Use fixed-point or deterministic math where needed.

26. **Vector assets are source of truth.** Any generated raster from a vector must be cacheable but never manually edited. The vector file is what gets versioned. If a raster needs tweaking, change the vector source.

27. **Game systems must be ling-lang first.** Implement rendering engine bindings, audio system, input handling, and asset pipeline in ling-lang wherever a ling-lang FFI exists. Only write host-language code when absolutely required by the underlying platform or graphics API.

28. **Build pipeline produces all targets.** The build system must output Windows .exe, macOS .app, Linux binaries, and Web (WASM+JS) in one command. No platform-specific build scripts. One build, many targets.
